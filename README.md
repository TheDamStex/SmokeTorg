# SmokeTorg

## 1. Опис проєкту
SmokeTorg — це настільна система для автоматизації роздрібної торгівлі у форматі **POS + склад + дисконтна система**.

Коротко:
- Призначення: облік продажів, приходу товарів, залишків і клієнтських карт.
- Цільова аудиторія: невеликі та середні магазини, що працюють з касою та складом.
- Основна мета: дати єдиний робочий інструмент для касира, менеджера та адміністратора.

---

## 2. Технології
- **.NET 8** (`net8.0-windows`)
- **WPF** + патерн **MVVM**
- **MySQL**
- **MySqlConnector**
- **Dependency Injection**: `Microsoft.Extensions.DependencyInjection`
- Архітектурний підхід: **Layered / Clean-like**
- Локальне зберігання налаштувань БД з шифруванням пароля через **DPAPI**

---

## 3. Архітектура
Проєкт поділено на логічні шари:

- `Domain` — сутності бізнес-домену та enum-и ролей/статусів.
- `Application` — інтерфейси та прикладні сервіси (auth, продажі, склад, користувачі тощо).
- `Infrastructure` — доступ до даних, SQL-скрипти, ініціалізація БД, збереження налаштувань.
- `Presentation` — WPF Views/ViewModels, вікна, діалоги, навігація.
- `Common` — базові класи MVVM, команди, логер, загальні утиліти.

Ключові принципи:
- Репозиторії використовуються через інтерфейси (`IProductRepository`, `IUserRepository`, ...).
- Весь склад об’єктів реєструється через DI-контейнер у `App.xaml.cs`.
- Підтримується перемикання типу зберігання через фабрику (`Json` / `Sql`), але у поточній конфігурації частина модулів працює через JSON-репозиторії, а користувачі — через MySQL.
- `DbInitializer` відповідає за перевірку підключення, створення БД та схеми.
- `SetupWizard` запускається на першому старті й проводить початкову конфігурацію.

## 4. Поточний стан проєкту
### Повністю стабільні модулі
- **Аутентифікація та авторизація**: вхід користувача, перевірка ролей, базове управління доступом до екранів.
- **Setup Wizard**: перевірка підключення до MySQL, створення БД та базової схеми, ініціалізація адміністратора.
- **Базовий POS-процес**: формування продажу, розрахунок підсумків, вибір клієнтської картки.

### Частково реалізовані
- **Складський блок**: облік рухів і залишків працює, але частина сценаріїв інвентаризації та аналітики потребує розширення.
- **Звіти**: існує сервісна основа, але функціонал розширеної комерційної звітності ще не завершений.
- **Керування користувачами**: ключові операції доступні, однак розширені політики безпеки не завершені.

### В розробці
- Розширена бонусна логіка.
- Експорт даних (CSV/Excel/PDF).
- Розширені управлінські звіти.
- Керовані міграції схеми БД.

### Технічний борг
- Гібридне зберігання даних (частина модулів у JSON, частина — у MySQL) ускладнює наскрізну цілісність.
- Відсутній повний механізм версіонованих міграцій.
- Немає деталізованого аудиту дій користувачів на рівні окремих бізнес-подій.
- Потребує формалізації стратегія транзакцій для багатокрокових операцій.

### Поточна стратегія зберігання та контроль цілісності
- **Де використовується JSON**: `Products`, `Categories`, `Suppliers`, `Customers`, `Sales`, `Purchases`, `StockItems`, `StockMovements`, `Settings`.
- **Де повністю MySQL**: `Users` (автентифікація та управління акаунтами), `SchemaInfo` (службова інформація схеми), ініціалізаційні SQL-скрипти.
- **Транзакції**: у поточній версії немає централізованої реалізації бізнес-транзакцій для багатотабличних сценаріїв.
- **Audit-лог**: повноцінний аудит дій користувачів ще не реалізований.

## 5. Архітектурні принципи (НЕ МОЖНА порушувати)
- Використовується **Clean-like підхід** із чітким розділенням відповідальностей між шарами.
- Шар **Domain не залежить від Infrastructure** або UI-компонентів.
- Усі репозиторії підключаються **лише через інтерфейси** Application-шару.
- **Dependency Injection обов’язковий** для сервісів, репозиторіїв, ViewModel та інфраструктурних компонентів.
- **Заборонено писати SQL у ViewModel** або інших Presentation-класах.
- **Заборонено використовувати кирилицю в C#-іменах** (класи, методи, властивості, поля, enum).

---

## 6. Життєвий цикл застосунку
1. **Старт App**: ініціалізується DI-контейнер, реєструються сервіси та репозиторії, піднімається інфраструктура застосунку.
2. **Перевірка конфігурації БД**: читаються локальні налаштування підключення, валідність конфігурації перевіряється перед входом у робочий режим.
3. **Setup Wizard**: якщо конфігурації немає або вона невалідна — запускається майстер для створення/перевірки БД і початкового адміністратора.
4. **Login**: після успішної ініціалізації виконується автентифікація користувача та завантаження його ролі.
5. **Побудова UI залежно від ролі**: головне меню та доступні модулі фільтруються відповідно до прав (`Admin`, `Manager`, `Seller/Cashier`).
6. **Робота з модулями**: користувач працює з POS, складом, закупівлями, клієнтами та налаштуваннями в межах своєї ролі.

## 8. Матриця ролей і доступів
| Роль | POS | Склад | Користувачі | Налаштування БД |
|------|-----|-------|-------------|------------------|
| `Admin` | Повний доступ: продажі, повернення (за сценарієм), робота з картками | Повний доступ: залишки, рухи, прихід, коригування | Повний доступ: створення/редагування/деактивація акаунтів, ролі | Повний доступ: перегляд і зміна параметрів, перевірка підключення |
| `Manager` | Операційний доступ до продажів і клієнтів у межах бізнес-процесу | Доступ до перегляду та операцій складу, крім критичних системних налаштувань | Обмежений або відсутній доступ до адміністрування акаунтів (залежно від політики) | Немає доступу до критичних системних параметрів |
| `Seller` / `Cashier` | Робочий доступ до POS, сканування, оформлення продажу | Переважно перегляд залишків/наявності (без адміністративних змін) | Немає доступу | Немає доступу |

> Примітка: точні права визначаються політиками ViewModel та прикладних сервісів; для комерційної редакції рекомендовано винести їх у централізований policy-механізм.

## 9. Перший запуск (Setup Wizard)
На першому запуску виконується майстер налаштування:

1. Налаштування підключення до MySQL (host/port/database/user/password).
2. Створення БД (якщо не існує).
3. Автоматичне створення таблиць та службової схеми.
4. Створення облікового запису адміністратора.
5. Збереження конфігурації для наступних запусків.

---

## 10. Структура бази даних
Основні таблиці:

- `Users` — користувачі системи, ролі, стан облікового запису, парольні хеші/сіль.
- `Products` — товари (SKU, штрихкод, ціни, мін. залишок, категорія).
- `Categories` — категорії товарів.
- `Suppliers` — постачальники.
- `Customers` — клієнти та дисконтні картки/бонусні параметри.
- `Sales` / `SaleItems` — документи продажу та позиції продажу.
- `Purchases` / `PurchaseItems` — документи закупівлі та позиції приходу.
- `SchemaInfo` — службова інформація про версію схеми БД.

Додатково використовуються таблиці `Settings`, `StockItems`, `StockMovements` для налаштувань і складського обліку.

---

## 11. Модель даних (Domain Overview)
- **Product** — товарна одиниця продажу/обліку: ідентифікація, ціноутворення, категоризація, контроль мінімального залишку.
- **Customer** — клієнт із дисконтним профілем: карта, тип знижки, бонусні параметри та історично значущі атрибути взаємодії.
- **User** — обліковий запис співробітника з роллю, безпековими атрибутами та станом доступу до системи.
- **Sale** — документ продажу, що агрегує склад позицій, суму, знижки, прив’язку до клієнта та автора операції.
- **Purchase** — документ приходу від постачальника для поповнення складу з фіксацією позицій та закупівельних параметрів.
- **StockMovement** — подія руху залишку (вхід/вихід/корекція), що забезпечує трасування змін складського стану.

---

## 12. Потік даних (Data Flow)
Базові ланцюжки обробки в поточній архітектурі:

- `POS → SalesService → ISaleRepository → JSON (sales.json)`
- `POS/Inventory operations → InventoryService → IStockRepository → JSON (stockitems.json, stockmovements.json)`
- `Login → AuthService → IUserRepository → MySQL (Users)`
- `SetupWizard → DbInitializer → SchemaInfo`
- `Налаштування БД → DbSettingsService → локальний JSON-конфіг + DPAPI`

Такий потік відображає поточну гібридну модель зберігання (JSON + MySQL) і є перехідним до повної SQL-стратегії.

---

## 13. Управління користувачами
- Створювати нові акаунти може користувач з роллю `Admin`.
- Доступ до налаштувань БД та системних параметрів має `Admin`.
- Перевірка ролей виконується у ViewModel-рівні (зокрема в головній моделі, керуванні користувачами та вікні налаштувань БД) та підтримується бізнес-сервісами.

---

## 14. Безпека
- Паролі зберігаються у вигляді хешів (**PBKDF2**) із сіллю.
- Пароль до БД у локальному конфігу шифрується через **DPAPI** (CurrentUser scope).
- Для SQL доступу використовуються параметризовані запити (через MySqlConnector), що знижує ризики SQL injection.

## 16. Обмеження поточної версії
- Немає повноцінних версіонованих міграцій БД (лише базова ініціалізація схеми).
- Немає **multi-tenant** моделі для ізоляції кількох незалежних бізнесів в одному інстансі.
- Немає офлайн-режиму з подальшою синхронізацією.
- Немає distributed locking для координації конкурентних операцій у розподіленому середовищі.
- Немає детального аудиту змін по сутностях і користувачах у форматі event trail.

## 19. Як запустити проєкт
1. Встановити **.NET 8 SDK**.
2. Підготувати доступний екземпляр **MySQL**.
3. У корені рішення виконати:
   ```bash
   dotnet build SmokeTorg/SmokeTorg.csproj
   dotnet run --project SmokeTorg/SmokeTorg.csproj
   ```
4. На першому запуску пройти **Setup Wizard**.

> Примітка: застосунок є WPF (`net8.0-windows`), для повноцінного запуску потрібне середовище Windows.

## 20. Важливо для подальшої розробки
### Для AI / розробників
- Не використовувати кирилицю в назвах C#-класів, властивостей, методів і полів.
- Усі UI-тексти мають бути українською.
- Дотримуватись MVVM строго.
- Не розміщувати бізнес-логіку у `code-behind`.
- Усі нові модулі підключати через DI.
- Не ламати існуючі інтерфейси без окремого плану сумісності.

---

## 21. Для AI та розробників
Обов’язкові правила внесення змін:
- **MVVM строго**: ViewModel відповідає за стан і поведінку UI, View — лише за відображення.
- **Жодної бізнес-логіки у `code-behind`**: дозволені тільки технічні UI-обробники, що не містять доменних рішень.
- **Усі нові модулі через DI**: без ручного створення сервісів у Presentation-шарі.
- **Зберігати backward compatibility**: зміни повинні бути сумісні з існуючими сценаріями та даними.
- **Не ламати публічні інтерфейси**: зміни контрактів можливі лише з планом міграції та версіонуванням.
